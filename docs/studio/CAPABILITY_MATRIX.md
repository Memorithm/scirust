# SciRust Studio capability matrix

Generated by hand from the actual registry/adapter code as of Phase 2A (see
`docs/studio/adr/0001-capability-registry.md` and
`docs/studio/RUNTIME_CONTRACT.md`); regenerating it automatically from
`scirust catalog --format json` plus a source-tree scan is future work, not
yet built — treat that gap honestly rather than implying this file is
machine-generated when it is not.

Columns, exactly as specified for this matrix:

- **Catalogued** — the model/crate exists in the source repository (module
  or crate is present and, where checked, compiles).
- **Descriptor** — a `scirust_studio_registry::CapabilityDescriptor` exists
  for it.
- **Adapter** — a real `scirust_studio_runtime::CapabilityAdapter`
  implementation exists and is registered in `all_adapters()`.
- **Scenario tested** — a `.scirust.toml` scenario for it is executed by an
  automated test.
- **Oracle tested** — the underlying model has an analytic-solution or
  conservation-law test *in its own crate* (independent of Studio).
- **CLI exposed** — reachable through `scirust catalog`/`scirust run`.
- **Desktop exposed** — reachable from a desktop application. **Always
  "No" at this stage** — no desktop shell exists yet (Phase 5+).

"Yes, per `scirust-sim`'s own tests (not independently re-verified this
pass)" means: the audit read the module's own doc comments and knows tests
exist (per `cargo test -p scirust-sim` passing in CI), but this Studio pass
did not open the source and independently confirm the oracle's correctness
the way it did for the five adapted models. That distinction matters — do
not read "Oracle tested: Yes" here as "Studio verified this," only as "the
owning crate verifies this."

**5 of 16 modules have a real Studio adapter.** The other 11 (plus 3 model
families within already-partially-adapted modules) are real, tested code
that Studio does not yet expose. Building their adapters is Phase 3 work.

## The rest of the workspace

`docs/studio/REPOSITORY_AUDIT.md` §10 already classifies the ~120 other
workspace members. Restated briefly here for this matrix's sake: none of
them have a `CapabilityDescriptor` or `CapabilityAdapter`, none are CLI- or
desktop-exposed through Studio, and most are explicitly recommended to stay
catalogue-only pending real API review — OT/ICS device discovery
(`scirust-mcp`, `scirust-discovery`), live trading (`scirust-trader` with
its `live` feature), GPU/CUDA backends (already license-gated), and the
relativity/TDI research crates whose own docs describe them as
experimental. Adapting any of them should start with reading their real
public API, per the audit's own rule, not this matrix.
